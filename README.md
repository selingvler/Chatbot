This project implements a retrieval augmented conversational AI system focused on Harry Potter. The chatbot combines semantic search, rule-based decision logic, large language models and GUI for usability.
•	Python is used and environment variables are managed using python-dotenv to securely handle API key.
•	SentenceTransformers "intfloat/e5-large-v2" is used to convert textual corpus into high-dimensional vector embeddings. 
•	FAISS is used to store and search embeddings. Inner product (IP) similarity and L2 normalization is used. FAISS enabled high-confidence direct answer without needing LLM. Direct answer from FAISS threshold is set 0.9. 
•	The system applies multi-stage decision pipeline. If similarity score is below the threshold twice, both before and after going agentic level, the system refuses the answer. If similarity score is higher than 0.9, the system returns a stored answer directly. For intermediate cases, agent decides whether to answer using retrieved contents, query Wikipedia, as for clarification or reject the question. If there is ambiguity it asks for clarification or the question is completely unrelated it refuses to answer with action of fallback. Wikipedia API is used for Harry Potter related questions not covered by the corpus. Wikipedia usage is explicitly tagged and separated from the corpus content, maintaining transparency. A separate answering function generates responses strictly constrained to retrieved content.
•	A Tinker based GUI is implemented to display chat history, accept user input, show system metadata (answer source or decision path).
•	All conversations are logged into a CSV file, including dialog ID, user question and system answer.
Flowchart
The process starts when the user enters a question through the desktop interface. The input is converted into a vector representation using a sentence embedding model. This embedding is then compared against the vector database using FAISS to retrieve the most relevant sentences and a similarity score. Based on this score, the system follows one of three paths: low-confidence questions are rejected, high-confidence questions are answered directly from the local corpus without calling a language model, and intermediate cases are passed to an agentic decision layer. The agent resolves ambiguities using conversation history and decides whether to answer using the local corpus, query Wikipedia for external knowledge, ask for clarification, or reject the question. Finally, the generated answer is displayed to the user through the interface, and both the question and answer are logged.
